이 가이드는 AI 에이전트/개발 도구에 Agent Skills를 통합하는 전체 흐름을 설명합니다. 핵심은 동일합니다: 스킬 탐색, 모델에 공개, 활성화 시 본문 로드, 세션 동안 컨텍스트 유지.

아키텍처가 달라도 통합 원리는 같습니다. 차이는 주로 아래 두 가지입니다.

- **스킬 저장 위치**: 로컬 에이전트는 파일시스템 스캔이 가능하지만, 클라우드/샌드박스 환경은 API/레지스트리/번들 방식이 필요합니다.
- **모델의 콘텐츠 접근 방식**: 모델이 파일을 읽을 수 있으면 `SKILL.md`를 직접 로드하고, 그렇지 않다면 전용 활성화 도구 또는 프롬프트 주입이 필요합니다.

**사전 지식**: [Agent Skills 명세](/specification) (`SKILL.md` 형식, frontmatter, 디렉터리 규칙)

## 핵심 원칙: 점진적 공개

스킬 호환 에이전트는 같은 3단계 로딩 전략을 따릅니다.

| 단계 | 로딩 내용 | 시점 | 토큰 비용 |
|------|-----------|------|-----------|
| 1. 카탈로그 | 이름 + 설명 | 세션 시작 | 스킬당 약 50~100 |
| 2. 지침 | `SKILL.md` 본문 | 스킬 활성화 시 | 권장 5000 미만 |
| 3. 리소스 | scripts/references/assets | 지침에서 참조 시 | 가변 |

즉, 모든 스킬 본문을 선로딩하지 않고 필요한 것만 로드해 기본 컨텍스트를 작게 유지합니다.

## 1단계: 스킬 탐색

세션 시작 시 사용 가능한 스킬을 찾아 메타데이터를 읽습니다.

### 스캔 경로

환경에 따라 다르지만 로컬 에이전트는 보통 다음 범위를 스캔합니다.

- **프로젝트 범위**: 현재 저장소 전용 스킬
- **사용자 범위**: 사용자 전역 스킬

또한 각 범위에서 클라이언트 전용 경로 + 공용 관례 경로를 함께 스캔하는 것이 좋습니다.

| 범위 | 경로 | 목적 |
|------|------|------|
| Project | `<project>/.<your-client>/skills/` | 클라이언트 전용 |
| Project | `<project>/.agents/skills/` | 클라이언트 간 호환 |
| User | `~/.<your-client>/skills/` | 클라이언트 전용 |
| User | `~/.agents/skills/` | 클라이언트 간 호환 |

`.agents/skills/`는 교차 클라이언트 공유 관례로 널리 사용됩니다.

<Note>
실무에서는 `.claude/skills/`, git 루트까지의 상위 디렉터리(모노레포), XDG config 경로, 사용자 지정 경로를 추가로 스캔하기도 합니다.
</Note>

### 탐색 대상

스킬 디렉터리 내에서 **`SKILL.md` 파일을 정확히 포함한 하위 디렉터리**를 스킬로 간주합니다.

```
~/.agents/skills/
├── pdf-processing/
│   ├── SKILL.md          ← 탐색됨
│   └── scripts/
│       └── extract.py
├── data-analysis/
│   └── SKILL.md          ← 탐색됨
└── README.md             ← 무시됨
```

권장 규칙:

- `.git/`, `node_modules/` 등 불필요 디렉터리 제외
- 필요 시 `.gitignore` 반영
- 최대 깊이/최대 디렉터리 수 제한으로 과도한 스캔 방지

### 이름 충돌 처리

동일한 `name`이 여러 개 발견되면 결정 규칙을 고정하세요.

- 공통 관례: **프로젝트 스킬이 사용자 스킬보다 우선**
- 같은 범위 내부 충돌: first-found 또는 last-found 중 하나를 고정
- 충돌 시 경고 로그 출력(섀도잉 안내)

### 신뢰(Trust) 고려

프로젝트 스킬은 불신 저장소에서 유입될 수 있으므로, 프로젝트 폴더가 신뢰 상태일 때만 로드하도록 게이트를 두는 것이 안전합니다.

### 클라우드/샌드박스 환경

로컬 파일시스템 접근이 없으므로 범위별 전략이 필요합니다.

- 프로젝트 스킬: 클론된 리포지토리 내부에서 함께 탐색 가능
- 사용자/조직 스킬: 외부 소스(설정 리포, URL/패키지, 웹 업로드 등)로 프로비저닝
- 내장 스킬: 배포 아티팩트에 정적 번들

## 2단계: `SKILL.md` 파싱

각 `SKILL.md`에서 frontmatter와 본문을 추출합니다.

### frontmatter 추출

`SKILL.md`는 `---`로 감싼 YAML frontmatter + 그 뒤 Markdown 본문으로 구성됩니다.

1. 파일 시작 `---`와 종료 `---` 찾기
2. YAML 파싱 (`name`, `description` 필수 + 선택 필드)
3. 종료 구분자 이후 텍스트를 본문으로 사용

### 잘못된 YAML 처리

다른 클라이언트에서 만든 스킬은 느슨한 YAML을 포함할 수 있습니다(예: 콜론이 있는 미인용 값). 인용 보정/블록 스칼라 변환 후 재시도 같은 fallback을 두면 호환성이 좋아집니다.

### 느슨한 검증 정책

가능하면 경고 후 로드하세요.

- 디렉터리명과 `name` 불일치: 경고 + 로드
- `name` 길이 초과: 경고 + 로드
- `description` 없음/빈값: 스킵 + 오류 로그
- YAML 완전 파싱 실패: 스킵 + 오류 로그

진단 정보는 로그/디버그 명령/UI에서 사용자에게 노출할 수 있게 저장하세요.

<Note>
명세는 `name`에 엄격한 제약을 둡니다. 위 정책은 교차 클라이언트 호환성 강화를 위한 실용적 완화입니다.
</Note>

### 저장할 데이터

최소 필드:

| 필드 | 설명 |
|------|------|
| `name` | frontmatter |
| `description` | frontmatter |
| `location` | `SKILL.md` 절대 경로 |

활성화 시 빠른 조회를 위해 `name` 키 맵으로 저장하세요.

본문은
- 탐색 시 메모리에 캐시(활성화 빠름) 또는
- 활성화 시점 재읽기(메모리 절약 + 파일 변경 반영)
중 하나를 선택하면 됩니다.

## 3단계: 모델에 스킬 카탈로그 공개

전체 본문은 로드하지 않고, 사용 가능한 스킬 목록만 모델에 제공합니다(점진적 공개 1단계).

### 카탈로그 구성

각 스킬에 `name`, `description`, 필요 시 `location`을 포함합니다(XML/JSON/목록 모두 가능).

```xml
<available_skills>
  <skill>
    <name>pdf-processing</name>
    <description>Extract PDF text, fill forms, merge files. Use when handling PDFs.</description>
    <location>/home/user/.agents/skills/pdf-processing/SKILL.md</location>
  </skill>
</available_skills>
```

`location`은 파일 기반 활성화와 상대 경로 해석에 유용합니다.

### 카탈로그 위치

일반적으로 두 방식이 쓰입니다.

- **시스템 프롬프트 섹션**: 가장 단순하고 폭넓게 호환
- **전용 활성화 도구 설명**: 프롬프트를 깔끔히 유지하고 활성화 로직과 결합

### 동작 지침

카탈로그 옆에 "언제/어떻게 스킬을 로드할지"를 짧게 안내합니다.

파일 읽기 방식:

```
The following skills provide specialized instructions for specific tasks.
When a task matches a skill's description, use your file-read tool to load
the SKILL.md at the listed location before proceeding.
When a skill references relative paths, resolve them against the skill's
directory (the parent of SKILL.md) and use absolute paths in tool calls.
```

전용 도구 방식:

```
The following skills provide specialized instructions for specific tasks.
When a task matches a skill's description, call the activate_skill tool
with the skill's name to load its full instructions.
```

### 필터링

비활성화/권한 거부/모델 호출 금지 플래그가 있는 스킬은 카탈로그에서 **완전히 숨기세요**. 보여주고 막는 방식은 모델 턴을 낭비합니다.

### 스킬이 없을 때

카탈로그와 관련 안내를 아예 생략하세요. 빈 블록이나 빈 옵션 도구 등록은 혼란을 줍니다.

## 4단계: 스킬 활성화

모델 또는 사용자가 스킬을 선택하면 전체 지침을 대화 컨텍스트에 주입합니다(점진적 공개 2단계).

### 모델 주도 활성화

대부분 구현은 하네스 키워드 매칭보다 모델 판단을 활용합니다.

- **파일 읽기 활성화**: 모델이 카탈로그의 `SKILL.md` 경로를 파일 읽기 도구로 직접 로드
- **전용 도구 활성화** (`activate_skill` 등): 이름 입력 → 스킬 콘텐츠 반환

전용 도구 장점:

- frontmatter 포함/제외 제어
- 태그 래핑으로 컨텍스트 관리 용이
- 번들 리소스 목록 동시 제공
- 권한/동의 제어
- 활성화 분석 로깅

<Tip>
전용 도구를 쓴다면 `name` 파라미터를 실제 스킬 이름 enum으로 제한하세요. 스킬이 없으면 도구 자체를 등록하지 않는 편이 안전합니다.
</Tip>

### 사용자 명시 활성화

사용자도 `/skill-name` 또는 `$skill-name` 같은 문법으로 직접 활성화할 수 있게 하세요. 자동완성 위젯을 붙이면 발견성이 좋아집니다.

### 모델에 전달할 콘텐츠

두 방식 모두 실무에서 사용됩니다.

- **전체 파일 전달**: frontmatter 포함
- **본문만 전달**: frontmatter 제거 후 Markdown만 반환

### 구조화 래핑

전용 도구라면 식별 태그 래핑을 권장합니다.

```xml
<skill_content name="pdf-processing">
# PDF Processing
...
Skill directory: /home/user/.agents/skills/pdf-processing
Relative paths in this skill are relative to the skill directory.

<skill_resources>
  <file>scripts/extract.py</file>
  <file>references/pdf-spec-summary.md</file>
</skill_resources>
</skill_content>
```

장점:

- 모델이 스킬 콘텐츠를 일반 대화와 구분 가능
- 컨텍스트 압축 시 보존 대상 식별 가능
- 리소스 목록 노출(지연 로딩 유지)

### 번들 리소스 목록

리소스는 목록만 제공하고, 실제 파일 내용은 필요 시 모델이 파일 읽기 도구로 개별 로드하도록 하세요.

### 권한 allowlist

파일 권한 게이트가 있는 에이전트라면 스킬 디렉터리를 allowlist에 추가해 참조 파일 접근 시 매번 승인 팝업이 뜨지 않게 하세요.

## 5단계: 세션 중 스킬 컨텍스트 관리

한 번 주입된 스킬 지침이 세션 내내 효과를 유지하도록 관리합니다.

### 컨텍스트 압축에서 스킬 콘텐츠 보호

메시지 요약/삭제 로직이 있다면 스킬 콘텐츠는 prune 대상에서 제외하세요. 중간에 지침이 사라지면 눈에 띄는 오류 없이 성능이 떨어집니다.

- 스킬 도구 출력에 보호 플래그 부여
- 구조화 태그를 기반으로 보존 대상 식별

### 중복 활성화 제거

세션 내 이미 활성화된 스킬은 재주입을 생략해 중복 지침 축적을 막을 수 있습니다.

### 서브에이전트 위임(선택)

고급 패턴으로, 본 대화에 지침을 넣지 않고 별도 서브에이전트 세션에서 스킬을 실행한 뒤 결과 요약만 메인 대화로 반환할 수 있습니다. 복잡한 스킬 워크플로에서 유용합니다.
