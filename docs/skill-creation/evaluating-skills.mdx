프롬프트 하나에서 잘 동작해 보이는 스킬도, 다양한 요청과 엣지 케이스에서 안정적으로 동작하는지는 별도 검증이 필요합니다. 구조화된 평가(evals)를 돌리면 이 질문에 답하고, 개선 루프를 만들 수 있습니다.

## 테스트 케이스 설계

테스트 케이스는 보통 3요소로 구성합니다.

- **Prompt**: 실제 사용자 문장
- **Expected output**: 성공 기준을 사람이 읽을 수 있게 기술
- **Input files**(선택): 스킬 실행에 필요한 파일

스킬 디렉터리의 `evals/evals.json`에 저장하세요.

```json evals/evals.json
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "I have a CSV of monthly sales data in data/sales_2025.csv. Can you find the top 3 months by revenue and make a bar chart?",
      "expected_output": "A bar chart image showing the top 3 months by revenue, with labeled axes and values.",
      "files": ["evals/files/sales_2025.csv"]
    }
  ]
}
```

좋은 테스트 프롬프트 팁:

- 처음엔 2~3개로 시작
- 표현 방식/상세도 다양화
- 엣지 케이스 최소 1개 포함
- 실제 맥락(파일 경로, 컬럼명, 사용자 맥락) 반영

초기에는 프롬프트+기대 출력까지만 만들고, 첫 실행 결과를 본 뒤 assertion을 추가해도 충분합니다.

## 평가 실행

핵심 패턴은 각 테스트를 최소 두 번 실행하는 것입니다.

- **with_skill**
- **without_skill**(또는 이전 버전 스킬)

이렇게 해야 스킬의 순수 효과를 비교할 수 있습니다.

### 워크스페이스 구조

각 반복(iteration)마다 별도 디렉터리를 두고, 테스트별로 `with_skill/`과 `without_skill/`를 분리합니다.

```
csv-analyzer/
├── SKILL.md
└── evals/
    └── evals.json
csv-analyzer-workspace/
└── iteration-1/
    ├── eval-top-months-chart/
    │   ├── with_skill/
    │   │   ├── outputs/
    │   │   ├── timing.json
    │   │   └── grading.json
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    └── benchmark.json
```

### 실행 시 주의점

각 실행은 **깨끗한 컨텍스트**에서 시작해야 합니다. 이전 대화 상태가 남으면 스킬 효과를 공정하게 비교하기 어렵습니다.

각 실행에 제공할 정보:

- 스킬 경로(또는 baseline이면 없음)
- 테스트 프롬프트
- 입력 파일
- 출력 디렉터리

### baseline 선택

기존 스킬 개선이라면 "스킬 없음"보다 **이전 버전 스냅샷**을 baseline으로 쓰는 것이 유의미합니다.

### timing 수집

품질뿐 아니라 비용(시간/토큰)도 함께 기록하세요.

```json timing.json
{
  "total_tokens": 84852,
  "duration_ms": 23332
}
```

## assertion 작성

assertion은 출력이 만족해야 할 검증 가능한 조건입니다. 첫 실행 결과를 보고 추가하면 더 현실적입니다.

좋은 assertion:

- "출력 파일이 유효한 JSON이다"
- "막대 차트 축 라벨이 있다"
- "추천 항목이 최소 3개다"

약한 assertion:

- "출력이 좋다"
- "정확히 이 문구를 포함한다"(과도하게 brittle)

```json evals/evals.json
{
  "evals": [
    {
      "id": 1,
      "assertions": [
        "The output includes a bar chart image file",
        "The chart shows exactly 3 months",
        "Both axes are labeled"
      ]
    }
  ]
}
```

## 결과 채점

각 assertion을 PASS/FAIL로 기록하고, 반드시 구체 증거를 남기세요.

```json grading.json
{
  "assertion_results": [
    {
      "text": "Both axes are labeled",
      "passed": false,
      "evidence": "Y-axis label exists, X-axis label missing"
    }
  ],
  "summary": {
    "passed": 3,
    "failed": 1,
    "total": 4,
    "pass_rate": 0.75
  }
}
```

코드로 검증 가능한 항목(JSON 유효성, 파일 존재, 행 수 계산 등)은 LLM 판단보다 스크립트 검증이 안정적입니다.

## 집계 및 비교

iteration 단위로 요약 지표를 `benchmark.json`에 저장하세요.

```json benchmark.json
{
  "run_summary": {
    "with_skill": {
      "pass_rate": { "mean": 0.83, "stddev": 0.06 },
      "time_seconds": { "mean": 45.0, "stddev": 12.0 },
      "tokens": { "mean": 3800, "stddev": 400 }
    },
    "without_skill": {
      "pass_rate": { "mean": 0.33, "stddev": 0.10 },
      "time_seconds": { "mean": 32.0, "stddev": 8.0 },
      "tokens": { "mean": 2100, "stddev": 300 }
    },
    "delta": {
      "pass_rate": 0.50,
      "time_seconds": 13.0,
      "tokens": 1700
    }
  }
}
```

`delta`는 "추가 비용(시간/토큰) 대비 품질 향상"을 보여줍니다.

<Note>
`stddev`는 반복 실행 수가 충분할 때 의미가 큽니다. 초기에는 pass count와 delta 위주로 보세요.
</Note>

## 패턴 분석

집계 수치만 보면 놓치는 것이 많습니다.

- 양쪽 모두 항상 PASS인 assertion: 정보량 낮으므로 제거/교체
- 양쪽 모두 항상 FAIL인 assertion: 난이도/검증 항목 자체 재검토
- with_skill만 PASS하는 assertion: 스킬이 실제 가치 주는 지점
- 실행 간 결과 변동이 큰 경우: 지침 모호성 또는 테스트 flaky 가능성 점검
- 시간/토큰 outlier: 실행 트레이스로 병목 확인

## 사람 리뷰

assertion은 작성한 항목만 검증합니다. 사람 리뷰는 예상 못 한 문제(방향성 미스, 형식 품질, 실용성 부족)를 잡아냅니다.

```json feedback.json
{
  "eval-top-months-chart": "축 라벨 누락, 월 정렬이 시간순이 아닌 알파벳순",
  "eval-clean-missing-emails": ""
}
```

구체 피드백이 있는 케이스에 개선 우선순위를 두세요.

## 스킬 반복 개선

개선 신호는 크게 3가지입니다.

- 실패한 assertion(명확한 결함)
- 사람 피드백(품질/구조 문제)
- 실행 트레이스(왜 실패했는지 원인)

이 신호들과 현재 `SKILL.md`를 함께 LLM에 주고 수정안을 받는 방식이 효율적입니다. 단, 아래 원칙을 지키세요.

- 테스트 케이스 맞춤 패치가 아니라 일반화 개선
- 규칙 과다 추가보다 간결한 핵심 지침 유지
- "왜"를 설명하는 지침 선호(단순 금지/강제보다 효과적)
- 반복되는 작업은 `scripts/`로 번들링

### 반복 루프

1. 신호 + 현재 `SKILL.md`로 개선안 생성
2. 변경 적용
3. 새 `iteration-<N+1>/`에서 전체 재실행
4. 채점 + 집계
5. 사람 리뷰

결과가 충분히 안정적이거나, 반복 개선 이득이 사라지면 종료합니다.

[`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator) 스킬은 eval 실행, assertion 채점, 벤치마크 집계, 리뷰 표시를 자동화해 이 과정을 크게 줄여줍니다.
</Tip>
