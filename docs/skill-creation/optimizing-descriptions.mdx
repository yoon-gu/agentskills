스킬은 활성화되어야만 도움이 됩니다. `SKILL.md` 프론트매터의 `description` 필드는 에이전트가 특정 작업에서 스킬을 로드할지 결정하는 핵심 신호입니다. 설명이 너무 약하면 필요할 때 트리거되지 않고, 너무 넓으면 불필요하게 트리거됩니다.

이 가이드는 스킬 설명의 트리거 정확도를 체계적으로 테스트하고 개선하는 방법을 설명합니다.

## 스킬 트리거 방식

에이전트는 컨텍스트 관리를 위해 [점진적 공개](/what-are-skills#스킬은-어떻게-동작하나요)를 사용합니다. 시작 시에는 각 스킬의 `name`과 `description`만 로드하고, 사용자 요청이 설명과 맞을 때만 전체 `SKILL.md`를 읽어 지침을 수행합니다.

즉, 트리거 관점에서 설명이 사실상 전부입니다. 설명이 스킬의 쓰임새를 제대로 전달하지 못하면 에이전트는 해당 스킬을 선택하지 않습니다.

중요한 뉘앙스도 있습니다. 에이전트는 보통 기본 능력만으로 해결 가능한 단순 작업에서는 스킬을 참조하지 않을 수 있습니다. 예를 들어 "이 PDF 읽어줘" 같은 단일 요청은 설명이 잘 맞아도 스킬 없이 처리될 수 있습니다. 반대로 익숙하지 않은 API, 도메인 특화 워크플로, 특수 포맷처럼 전문성이 필요한 과제에서는 설명 품질이 큰 차이를 만듭니다.

## 효과적인 설명 작성

테스트 전에, 좋은 설명의 기준을 먼저 잡는 것이 좋습니다.

- **명령형으로 작성**: "이 스킬은..."보다 "다음 상황에서 이 스킬을 사용하라..."처럼 에이전트의 행동 판단을 직접 유도하세요.
- **구현이 아니라 의도 중심**: 내부 동작보다 사용자가 달성하려는 목표를 설명하세요.
- **적용 상황을 적극적으로 명시**: 사용자가 도메인 키워드를 직접 쓰지 않아도 적용되는 경우까지 포함하세요.
- **간결하게 유지**: 몇 문장~짧은 문단이 적절합니다. 너무 길면 여러 스킬에서 컨텍스트 부담이 커집니다. [명세](/specification#description-필드)상 최대 1024자 제한이 있습니다.

## 트리거 평가 질의 설계

트리거 테스트를 위해서는 현실적인 사용자 프롬프트와 기대 결과(`should_trigger`)를 라벨링한 eval 질의 세트가 필요합니다.

```json eval_queries.json
[
  { "query": "I've got a spreadsheet in ~/data/q4_results.xlsx with revenue in col C and expenses in col D — can you add a profit margin column and highlight anything under 10%?", "should_trigger": true },
  { "query": "whats the quickest way to convert this json file to yaml", "should_trigger": false }
]
```

보통 20개 정도를 권장합니다: 트리거되어야 하는 질의 8~10개, 트리거되면 안 되는 질의 8~10개.

### 트리거되어야 하는 질의

설명이 스킬 범위를 제대로 포착하는지 확인합니다. 아래 축으로 다양성을 주세요.

- **표현 방식**: 격식체, 구어체, 오타/약어 포함
- **명시성**: 도메인을 직접 언급하는 경우와 우회적으로 요구를 표현하는 경우
- **상세도**: 짧은 요청과 맥락이 긴 요청 혼합
- **복잡도**: 단일 단계와 다단계 작업 혼합

특히 유용한 케이스는, 스킬이 도움되지만 질의만 봐서는 연결이 즉시 드러나지 않는 경우입니다. 이런 케이스에서 설명 문구의 품질이 실제 성능 차이를 만듭니다.

### 트리거되면 안 되는 질의

가치가 큰 부정 테스트는 **근접 오답(near-miss)**입니다. 즉, 키워드는 유사하지만 실제로는 다른 작업인 케이스입니다.

CSV 분석 스킬 기준 예시:

- 약한 부정 예시: `"Write a fibonacci function"` (너무 무관)
- 약한 부정 예시: `"What's the weather today?"` (키워드 겹침 없음)
- 강한 부정 예시: `"I need to update the formulas in my Excel budget spreadsheet"` (스프레드시트 맥락은 같지만 CSV 분석이 아님)
- 강한 부정 예시: `"can you write a python script that reads a csv and uploads each row to our postgres database"` (CSV는 등장하지만 과업은 ETL)

### 현실성 높이기 팁

실사용 프롬프트를 반영하세요.

- 파일 경로 (`~/Downloads/report_final_v2.xlsx`)
- 개인 맥락 (`"my manager asked me to..."`)
- 구체 세부정보 (열 이름, 회사명, 값)
- 구어체, 약어, 간헐적 오타

## 설명이 실제로 트리거되는지 테스트하기

기본 방법은 간단합니다. 스킬을 설치한 상태에서 각 질의를 실행하고, 에이전트가 해당 스킬을 호출했는지 관찰하세요. 스킬 등록/탐색 방식은 클라이언트마다 다릅니다(스킬 디렉터리, 설정 파일, CLI 플래그 등).

대부분의 에이전트 클라이언트는 실행 로그, 도구 호출 이력, verbose 출력 같은 관찰 수단을 제공합니다. 스킬의 `SKILL.md`를 로드했다면 트리거된 것입니다.

질의는 다음일 때 "통과"입니다.
- `should_trigger`가 `true`이고 스킬이 호출됨
- `should_trigger`가 `false`이고 스킬이 호출되지 않음

### 여러 번 실행

모델 동작은 비결정적입니다. 같은 질의도 실행마다 결과가 다를 수 있으므로, 각 질의를 여러 번(권장 3회) 실행해 **trigger rate**(호출 비율)를 계산하세요.

- should-trigger 질의: trigger rate가 임계값(기본 0.5 권장) 이상이면 통과
- should-not-trigger 질의: trigger rate가 임계값 미만이면 통과

질의 20개 x 3회면 60회 실행이므로 스크립트 자동화를 권장합니다. 아래는 일반 구조 예시입니다. `claude` 호출 및 `check_triggered` 감지 로직은 사용 중인 클라이언트에 맞게 바꾸세요.

```bash
#!/bin/bash
QUERIES_FILE="${1:?Usage: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# This example uses Claude Code's JSON output to check for Skill tool calls.
# Replace this function with detection logic for your agent client.
# Should return 0 (success) if the skill was invoked, 1 otherwise.
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

<Tip>
클라이언트가 지원한다면 결과가 확정되는 시점(스킬을 호출했거나, 호출 없이 다른 작업을 시작한 시점)에 조기 종료해 시간과 비용을 줄일 수 있습니다.
</Tip>

## 과적합 방지: train/validation 분리

모든 질의에 맞춰 설명을 최적화하면 과적합 위험이 있습니다. 특정 문구에는 잘 맞지만 새 질의에서는 실패할 수 있습니다.

질의 세트를 분리하세요.

- **Train 세트(~60%)**: 실패 원인을 찾고 설명 개선에 사용
- **Validation 세트(~40%)**: 일반화 성능 확인용으로 분리 보관

두 세트 모두 should-trigger/should-not-trigger 비율을 비슷하게 유지하세요. 반복 실험 간 비교 가능하도록 무작위 셔플 후 분할은 고정하는 것이 좋습니다.

위와 같은 스크립트를 쓴다면 `train_queries.json`과 `validation_queries.json`으로 분리해 각각 실행하면 됩니다.

## 최적화 루프

1. 현재 설명을 *train/validation 모두에서 평가*합니다.
2. *train 세트* 실패를 분석합니다.
   - 트리거되어야 하는데 안 된 질의
   - 트리거되면 안 되는데 된 질의
3. 설명을 수정합니다.
   - should-trigger 실패가 많으면: 범위가 너무 좁을 수 있으니 적용 맥락을 확장
   - should-not-trigger 오탐이 많으면: 범위가 너무 넓을 수 있으니 경계를 더 명확히 기술
   - 실패 질의의 키워드를 그대로 추가하지 말고, 그 질의가 대표하는 일반 범주를 설명에 반영
   - 여러 번 막히면 미세 조정보다 문장 구조/프레이밍을 크게 바꿔보기
   - 1024자 제한을 계속 확인
4. 1~3을 반복해 train 세트 통과율이 충분해질 때까지 개선합니다.
5. 최종 선택은 *validation 통과율*로 결정합니다. 가장 좋은 설명이 반드시 마지막 버전일 필요는 없습니다.

보통 5회 내외 반복이면 경향이 드러납니다. 성능이 전혀 개선되지 않으면 설명보다 질의 세트 품질(난이도, 라벨링, 현실성)을 먼저 점검하세요.

<Tip>
[`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator) 스킬은 이 루프를 자동화합니다. eval 세트 분할, 병렬 평가, 설명 개선 제안, HTML 리포트 생성까지 한 번에 처리할 수 있습니다.
</Tip>

## 결과 적용

최적 설명을 고른 뒤 다음을 수행하세요.

1. `SKILL.md` 프론트매터의 `description` 필드를 업데이트합니다.
2. [1024자 제한](/specification#description-필드) 이내인지 확인합니다.
3. 예상대로 트리거되는지 검증합니다. 수동 sanity check 후, 최적화에 쓰지 않은 새 질의 5~10개로 재평가하면 일반화 확인에 유리합니다.

Before and after:

```yaml
# Before
description: Process CSV files.

# After
description: >
  Analyze CSV and tabular data files — compute summary statistics,
  add derived columns, generate charts, and clean messy data. Use this
  skill when the user has a CSV, TSV, or Excel file and wants to
  explore, transform, or visualize the data, even if they don't
  explicitly mention "CSV" or "analysis."
```

개선된 설명은 스킬이 하는 일(요약 통계, 파생 컬럼, 차트, 정제)을 더 구체적으로 제시하면서, 적용 범위(CSV/TSV/Excel 및 비명시적 질의)도 더 넓게 포착합니다.
