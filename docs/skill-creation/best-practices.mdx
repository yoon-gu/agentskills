## 실제 전문성에서 시작하기

스킬 작성에서 흔한 실수는 도메인 맥락 없이 LLM의 일반 지식만으로 스킬을 만들게 하는 것입니다. 그러면 "오류를 적절히 처리하라" 같은 추상적 문구만 남고, 실제로 가치가 있는 API 패턴/엣지 케이스/팀 규칙이 빠집니다.

좋은 스킬은 실제 전문성에 기반합니다. 핵심은 프로젝트 고유 맥락을 생성 과정에 넣는 것입니다.

### 실전 작업에서 패턴 추출

에이전트와 실제 업무를 수행하면서 맥락, 교정, 선호를 제공한 뒤 재사용 가능한 부분을 스킬로 추출하세요.

- **성공한 단계**: 실제로 효과가 있었던 순서
- **교정한 지점**: 접근을 바꿔 준 부분(예: 라이브러리 선택, 특정 엣지 케이스 확인)
- **입출력 형태**: 입력/출력 데이터 구조
- **프로젝트 맥락**: 에이전트가 원래 모르는 제약/관례

### 기존 산출물에서 합성

이미 축적된 지식이 있다면 LLM에 제공해 스킬로 합성할 수 있습니다. 다만 일반 자료보다 팀의 실제 사고 보고서, 런북, 코드 리뷰 히스토리처럼 **현장 데이터**가 훨씬 유용합니다.

좋은 소스:

- 내부 문서/런북/스타일 가이드
- API 명세/스키마/설정 파일
- 코드 리뷰 코멘트/이슈 트래커
- 버전 관리 히스토리(특히 실제 수정 패치)
- 실제 장애 사례와 해결 방법

## 실제 실행으로 개선하기

초안 스킬은 대개 개선이 필요합니다. 실작업에 적용하고 결과(실패뿐 아니라 성공도)를 다시 반영하세요.

- 어떤 상황에서 오탐이 났는가?
- 어떤 요청을 놓쳤는가?
- 무엇을 덜어낼 수 있는가?

복잡한 도메인일수록 실행-수정 루프를 여러 번 도는 것이 효과적입니다.

<Tip>
최종 출력만 보지 말고 실행 트레이스를 함께 보세요. 불필요한 우회가 많다면 지침이 모호하거나, 현재 작업에 맞지 않는 지침이 포함되었거나, 기본값 없는 선택지가 과도할 가능성이 큽니다.
</Tip>

체계적인 반복 방법은 [스킬 출력 품질 평가](/skill-creation/evaluating-skills)를 참고하세요.

## 컨텍스트 예산을 아끼기

스킬이 활성화되면 `SKILL.md` 본문 전체가 대화 이력/시스템 컨텍스트/다른 스킬과 함께 로드됩니다. 즉, 스킬의 모든 토큰은 주의 자원을 경쟁합니다.

### 에이전트가 모르는 것만 넣기

프로젝트 특화 규칙, 도메인 특화 절차, 비자명 엣지 케이스처럼 스킬 없이는 알기 어려운 정보에 집중하세요.

````markdown
<!-- 장황한 예시 -->
## Extract PDF text
PDF (Portable Document Format) files are a common file format...

<!-- 더 나은 예시 -->
## Extract PDF text
Use pdfplumber for text extraction. For scanned documents, fall back to
pdf2image with pytesseract.
````

각 문장마다 "이 지침이 없으면 에이전트가 틀릴까?"를 점검하세요. 아니라면 과감히 줄이세요.

### 응집도 있는 범위 설계

스킬 범위 결정은 함수 설계와 비슷합니다. 너무 좁으면 여러 스킬을 동시에 로드해야 하고, 너무 넓으면 정확한 활성화가 어려워집니다.

### 중간 수준의 상세도

지나치게 포괄적인 스킬은 오히려 성능을 해칠 수 있습니다. 핵심 절차 + 작동 예시 중심의 간결한 구성이 일반적으로 더 강합니다.

### 대형 스킬은 점진적 공개로 분리

[명세](/specification#점진적-공개)는 `SKILL.md`를 500줄/5,000토큰 이하로 유지할 것을 권장합니다. 상세 자료는 `references/` 같은 별도 파일로 옮기고, **언제 읽어야 하는지 조건**을 명시하세요.

예:
- 나쁨: "`references/`를 참고해라"
- 좋음: "API가 non-200이면 `references/api-errors.md`를 읽어라"

## 제어 강도 보정

모든 지침을 같은 강도로 강제할 필요는 없습니다. 작업의 취약성과 일관성 요구에 맞춰 조절하세요.

### 취약도에 맞춰 구체성 조절

여러 방법이 허용되는 작업은 자유도를 주고, 순서/정합성이 중요한 작업은 엄격하게 지시하세요.

```markdown
## Code review process
1. Check SQL injection
2. Verify auth checks
3. Look for race conditions
4. Confirm no sensitive leak
```

````markdown
## Database migration
Run exactly this sequence:
```bash
python scripts/migrate.py --verify --backup
```
Do not modify the command.
````

### 메뉴 대신 기본값 제공

여러 도구가 가능하더라도 기본값을 하나 정하고 대안은 짧게 덧붙이세요.

### 선언보다 절차 중심

스킬은 특정 답 하나보다 문제군을 푸는 **방법**을 가르쳐야 재사용성이 높습니다.

```markdown
1. Read schema from references/schema.yaml
2. Join tables via _id conventions
3. Apply user filters as WHERE
4. Aggregate and format results
```

## 효과적인 지침 패턴

### Gotchas 섹션

가치가 큰 콘텐츠는 "에이전트가 합리적으로 오해할 지점"을 바로잡는 구체적 사실입니다.

````markdown
## Gotchas
- users 테이블은 soft delete 사용: `deleted_at IS NULL` 필수
- 같은 사용자 식별자가 시스템마다 `user_id`/`uid`/`accountId`로 다름
- `/health`는 웹 서버만 확인, 전체 상태는 `/ready` 확인
````

<Tip>
교정이 반복되는 오류는 바로 gotchas에 추가하세요. 가장 빠른 개선 루프입니다.
</Tip>

### 출력 템플릿

정해진 출력 형식이 필요하면 서술보다 템플릿이 더 안정적입니다. 짧은 템플릿은 `SKILL.md`, 긴 템플릿은 `assets/`에 두고 필요 시 로드하세요.

### 체크리스트

의존 단계가 있는 워크플로에서는 체크리스트가 누락을 크게 줄입니다.

```markdown
Progress:
- [ ] Analyze
- [ ] Map fields
- [ ] Validate
- [ ] Execute
- [ ] Verify
```

### 검증 루프

"작업 -> 검증 -> 수정 -> 재검증" 루프를 명시하세요.

```markdown
1. Make edits
2. Run validator
3. Fix errors
4. Re-run validator until pass
```

### 계획-검증-실행

대량/파괴적 작업은 중간 계획 산출물을 먼저 만들고, 소스 오브 트루스로 검증한 뒤 실행하세요.

```markdown
1. Analyze inputs -> form_fields.json
2. Build plan -> field_values.json
3. Validate plan
4. Fix and re-validate
5. Execute
```

핵심은 3단계의 자동 검증입니다. 검증기가 구체 오류를 반환해야 에이전트가 자체 수정할 수 있습니다.

### 재사용 스크립트 번들링

[스킬 반복 개선](/skill-creation/evaluating-skills) 중 실행 트레이스를 비교했을 때, 에이전트가 매번 같은 로직을 재구현한다면 `scripts/`에 검증된 스크립트로 고정하세요.

스크립트 설계/번들링 자세한 내용은 [스킬에서 스크립트 사용하기](/skill-creation/using-scripts)를 참고하세요.
